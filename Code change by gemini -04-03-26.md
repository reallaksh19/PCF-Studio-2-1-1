# Code changes by Gemini (04-03-2026)

## Issue Context: Geometric Gap Closure & Coordinate Drift

We resolved an ongoing issue where PCF pipeline coordinates (specifically `EP1` and `EP2` endpoints) were not precisely connecting mathematically, often leaving micro-gaps (e.g., 3.0mm) between adjacent components like flanges and pipes in the final output text or 3D viewer. Although the gaps fell cleanly beneath the user's Continuity Tolerance setting, the application incorrectly forced the original un-snapped GUI data into the output.

We aggressively addressed this across the Geometry Engine (Stage 5) and the Text Generator (Phase 2), ensuring a fully airtight geometric output.

---

### Fix 1: Stage 5 Universal "Local Stretch" Geometry (sequencer.js)

**Reasoning:**
The application had an algorithmic "gap-closer," but it only correctly worked under specific mapping paradigms. Furthermore, the Sequencer builds the structural array (`ordered`) of components. 

To guarantee that pipeline pieces form completely sealed mathematical bonds, we built an `applyLocalStretch` engine at the very end of Stage 5 (`runSequencer`). Once the routing algorithms definitively choose the pipeline sequence array, this function sweeps the array from front to back, checks the true physical distance between every neighboring component, and stretches `EP2` of the preceding component cleanly onto `EP1` of the next component. 

We injected this helper structurally at the bottom of the script, guaranteeing perfectly sealed geometry across all architectures: **Chain-Based**, **Sequential**, and **Strict Graph** modes.

**Code Snippet (`js/graph/sequencer.js`):**
```javascript
  // ── Universal Stage 5 P2 Gap Closure (Local Stretch) ──────────────
  // Now that a sequence sequence is definitively locked in, we iterate through
  // the ordered groups and physically stretch the P2 of the previous component
  // forward to seamlessly touch the P1 of the current component.
  const applyLocalStretch = (orderedList) => {
    for (let i = 1; i < orderedList.length; i++) {
      const prevRef = orderedList[i - 1];
      const currRef = orderedList[i];
      const prevG = groups.get(prevRef);
      const currG = groups.get(currRef);

      if (prevG?.pts?.['2'] && currG?.pts?.['1']) {
        const pEP2 = prevG.pts['2'];
        const cEP1 = currG.pts['1'];
        if (pEP2.E !== undefined && cEP1.E !== undefined) {
          const dx = cEP1.E - pEP2.E;
          const dy = cEP1.N - pEP2.N;
          const dz = cEP1.U - pEP2.U;
          const gap = Math.sqrt(dx * dx + dy * dy + dz * dz);

          // If it's a micro-gap within tolerance, literally stretch P2 forward
          // We ignore Sequence Number boundaries here because if they are routed adjacent,
          // they physically touch.
          if (gap > 0.001 && gap <= tolerance) {
            prevG.pts['2'] = { ...pEP2, E: cEP1.E, N: cEP1.N, U: cEP1.U };
            info(MOD, 'runSequencer', `Stage 5: Gap-filled ${prevRef}→${currRef} (${gap.toFixed(2)}mm) by stretching P2.`);
          }
        }
      }
    }
  };
```

*(Note: We then called `applyLocalStretch(ordered);` immediately preceding the `return` boundaries inside each mode's main block.)*

### Fix 2: Bypassing "Seq No" Artificial Barriers (sequencer.js)

**Reasoning:**
While debugging why the gap closer failed on a visible 3.0mm distance between Flange 1668 and 1670, we discovered that `Chain-Based Mode` had an original rule built to restrict gap filling exclusively to components sharing identically typed `Seq No.` values in the Table (`prevSeq === currSeq`). Because the user had components mapped to e.g., Sequence 11 and Sequence 15, the algorithm completely aborted sealing the physical 3.0mm fracture logic.

By ripping out this arbitrary string-comparison, the gap closure now evaluates the purest truth—*if you are mathematically routed next to a component, and the physical vector space between you is beneath the Continuity Tolerance (e.g., `< 25mm`), you will be snapped.* 

**Code Snippet Removed (`js/graph/sequencer.js` internal `_walkChain` block):**
```javascript
// DEL: if (gap > 0 && gap <= tolerance && prevSeq && prevSeq === currSeq) {
```

### Fix 3: Stopping Phase 2 PCF Coordinate Re-wipes (TableRegenerator.js)

**Reasoning:**
Even after `sequencer.js` successfully created airtight geometry natively in Stage 5, the PCF file output (Phase 2) was bizarrely returning to its broken 17986 coordinate drift. We traced this to `TableRegenerator.js` specifically overwriting mathematically stable `group.pts['1']` arrays with brute-forced numbers pulled straight from the raw text columns (`Start X/Y/Z`) just milliseconds before compiling the file.

We refactored `TableRegenerator` to immediately prioritize reading the `comp._group.pts` array (which holds all topologically stretched gap closures generated by the engine) rather than calculating arbitrary coordinates based on table strings.

**Code Snippet (`js/ui/table/TableRegenerator.js`):**
```javascript
// P2 Calculation: Prioritize True Mathematical Snapping from Topology Engine!
// DO NOT indiscriminately recalculate P2 using Grp L1/Axis 1 from the table row, 
// as this completely overwrites the 'Local Stretch' mathematically bridged gaps!
if (p1) {
    const existingP2Bore = comp.points[1]?.bore || comp.points[0]?.bore || 0;
    const grpPts = comp._group?.pts || {};
    const rawP2 = grpPts['2'];

    if (rawP2 && rawP2.E !== undefined) {
        // Topology engine provided a mathematically perfect P2 coordinate. USE IT!
        const p2 = { x: rawP2.E, y: rawP2.N, z: rawP2.U, bore: rawP2.bore || existingP2Bore };
        if (comp.points.length < 2) comp.points.push(p2);
        else comp.points[1] = p2;
    } else {
        // Fallback Vector Math... 
    }
}
```
