# CONSTRAINED CHAIN WALKER — Two-Pass Architecture

## Document ID: PCF-SF-003 Rev.0
## Supersedes: PCF-SF-002 (Single-pass Relaxed Walker)

---

## 1. PURPOSE AND PHILOSOPHY

The original Smart Fixer attempted to resolve all gaps and overlaps in a single massive pass. While effective for simple pipelines, in dense systems with missing data (like vents, OLETs, and supports), the "relaxed" topology matching sometimes improperly connected unrelated components.

The **Two-Pass Constrained Chain Walker** separates the logic into two distinct phases to strictly enforce engineering intent.

---

## 2. THE TWO-PASS ARCHITECTURE

### 2.1 First Pass: The Pipe Targeter
**Goal:** Fix the major physical gaps in the main headers by extending/trimming PIPE elements.
**Logic:**
- **Target Restriction:** Only generates fixes where at least one of the components in the gap is a `PIPE`. (Only pipes can stretch).
- **Bore Ratio Constraint:** `0.7 to 1.5`. A 400mm pipe will not stretch to connect to a 50mm OLET.
- **Line_Key Constraint:** If a Line_Key is mapped and present, it absolutely prohibits cross-line gap filling. (If Line_Key is missing, this check is safely skipped).
- **Single-Axis Constraint:** Gap filling/trimming is strictly enforced to happen along a primarily single axis. If a gap requires a multi-axis diagonal stretch (> 25mm drift), it is rejected as a data error.

*Why this works:* It solves 70% of the visible breaks in the 3D model safely, without hallucinating non-existent complex geometry.

### 2.2 Second Pass: The Non-Pipe Fitter
**Goal:** Address the remaining gaps between rigid fittings (Flange-to-Flange, Valve-to-Flange, etc.).
**Trigger:** Executed ONLY after the First Pass fixes have been approved by the user, applied to the Data Table, and the 3D view is refreshed.
**Logic:**
- Runs the same advanced rules (Bore Ratio, Line_Key restriction).
- Target Restriction: Specifically targets Non-Pipe components.
- Output: Because fittings are rigid (catalog dimensions), it generates "Add Node" or "Insert Pipe" fixing actions rather than stretching the component.

---

## 3. AUTO-APPROVAL TIERS

To streamline the user experience, fixes generated by the Orchestrator are pre-categorized:

1. **Auto-Approved (Tier 1/2):** Gaps `< 25.0mm` (Configurable `Continuity Tolerance`). These are extremely likely to be rounding errors, gasket omissions, or simple modeling drift. The UI marks them as `[Fix approved]`. The user can instantly hit "Approve All".
2. **Flagged for Review (Tier 3):** Gaps `> 25.0mm` and `< 20000.0mm`. Kept visible for the user to manually verify and click "Approve".
3. **Auto-Rejected (Tier 4):** Gaps `> 20000.0mm` (Configurable `Max Segment Length`). It is physically impossible to have a 20m implicit pipe gap in standard plant design; this indicates a severe disjoint in the line or a missing file segment. The UI marks them as `REJECT:`.

---

## 4. INTEGRATION WITH UI

1. User clicks **[1️⃣ Run First Pass]** in the header → Runs First Pass. UI shows the proposed pipe stretches.
2. User reviews Auto-Approved fixes or modifies approvals manually.
3. User clicks **[Apply Fixes & Update PCF]** → EPs of `PIPE` components are stretched or shortened. 3D model re-renders.
4. **[🚀 Run Second Pass]** button becomes visible in the header.
5. User clicks **[🚀 Run Second Pass]** → Analyzes rigid fitting connections.
6. User reviews Auto-Approved fixes or modifies approvals manually.
7. User clicks **[Apply Fixes & Update PCF]** → Finalizes the 3D model geometry.

## 5. CONFIGURATION OPTIONS

Several constraints are configurable in `validator-config.js` to tailor the logic:
- `minBoreRatio` / `maxBoreRatio` (Default: `0.7` to `1.5`): Ensures pipes only stretch to connect with compatible bore sizes.
- `minRadiiMultiplier` (Default: `0.2`): The minimum radius for an OLET offset is `0.2 * Nominal Bore`.
- `maxRadii` (Default: `13000.0mm`): The absolute maximum radius allowed for an OLET offset calculation.
